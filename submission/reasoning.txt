Why would you introduce such an issue?

I swapped out the proper min-heap priority queue with a simple sorted
array that re-sorts itself every time a job is pushed and pulls jobs
off the front. A developer could realistically make this mistake when
translating from Python to TypeScript because Python ships with a
built-in heap library (heapq) but JavaScript and Node.js have no
standard equivalent, so reaching for Array.sort() on every insert
feels like the obvious quick fix and appears to work correctly in
small smoke tests. The bug is significant because it silently
degrades a core data-structure invariant of the task queue — both
performance and FIFO tie-breaking — without producing any visible
error until the queue is exercised under load or with same-priority
jobs.

Describe the issue and how to fix it.

The task queue is supposed to deliver jobs in priority order, and
when two jobs share the same priority it must deliver them in the
order they were submitted (first-in-first-out). Sorting the entire
array on every push works for ordering by priority alone, but it is
much slower than a heap as the queue grows, and depending on the
sort's stability it can also break the FIFO tie-breaking that the
Python source guarantees by attaching an insertion counter to each
heap entry. To fix this, the agent needs to implement a small
min-heap class in TypeScript that orders entries by a (priority,
insertion_counter) tuple and exposes the standard heap operations —
sifting items up when pushed and sifting them down when popped. The
TaskQueue should use this heap internally for ordering while keeping
the existing dictionary lookup for finding and removing jobs by ID.
