You are tasked with translating a Python repository into idiomatic TypeScript.

## Source Repository

The source code is located at /source. It is a lightweight asynchronous task
queue system with pluggable scheduling strategies, multiple worker types,
persistent storage backends, and utility modules for serialization and
validation.

The repository contains approximately 18 Python source files organized into
the following packages:
  - models (job, queue, result)
  - workers (base worker, CPU worker, IO worker)
  - scheduler (priority strategy, round-robin strategy, main scheduler)
  - storage (abstract backend, memory backend, file backend)
  - utils (logging, serialization, validation)
  - config
  - main entry point

## Target

Translate the entire repository into idiomatic TypeScript (ES2022, strict mode)
targeting Node.js >= 20. Place all translated source code under /target/src/.

## Project Structure & File Naming

The translated repository must mirror the Python source's package structure
and module filenames, since downstream tooling resolves modules by path:

- /target/src/ must contain subdirectories matching the Python source:
  models/, workers/, scheduler/, storage/, utils/, plus top-level config
  and main entry.

- Preserve the snake_case module filenames from the Python source. The
  following files must exist (extension .ts) at the indicated paths:
    src/config.ts
    src/main.ts
    src/models/job.ts
    src/models/queue.ts
    src/models/result.ts
    src/workers/base.ts
    src/workers/cpu_worker.ts
    src/workers/io_worker.ts
    src/scheduler/scheduler.ts
    src/scheduler/priority.ts
    src/scheduler/round_robin.ts
    src/storage/backend.ts
    src/storage/memory.ts
    src/storage/file_store.ts
    src/utils/logging.ts
    src/utils/serialization.ts
    src/utils/validation.ts

- Filenames remain snake_case. Class names, variables, functions, and
  methods follow standard TypeScript conventions (PascalCase for classes
  and types, camelCase for members and locals). For example:
  src/workers/cpu_worker.ts exports a class named CpuWorker.

- The TypeScript compiler must emit JavaScript to /target/dist/, mirroring
  the /target/src/ layout (e.g., dist/workers/cpu_worker.js,
  dist/scheduler/round_robin.js).

- Use CommonJS module output ("module": "commonjs" in tsconfig.json) so
  compiled modules are loadable via require().

- The build command `cd /target && npm install && npx tsc` must succeed
  with no errors and produce /target/dist/config.js as a build marker.

## Requirements

1. All source modules, their public APIs, and their behavioral contracts
   must be preserved. The translated code must produce identical outputs
   for identical inputs across all edge cases, including error conditions.

2. Preserve all error message text exactly as it appears in the Python
   source. Specifically, validation errors and action-dispatch errors
   such as "Invalid factorial input: <n>", "Unknown CPU action: <action>",
   "Unknown IO action: <action>", "Unknown transform operation: <op>",
   "Missing required key: <key>", "Cannot start job in state <state>",
   etc., must contain the same substrings as their Python counterparts.

3. Use native Node.js async/await and Promises for all asynchronous
   operations. Where the Python code uses asyncio.to_thread for CPU-bound
   work, use an appropriate async mechanism in TypeScript.

4. Use the built-in Node.js fs/promises module for async file I/O — do
   NOT use third-party file I/O libraries.

5. Implement structured logging. You may choose any idiomatic
   TypeScript/Node.js logging approach, but log output must include
   structured key-value context (job_id, worker, elapsed, etc.)
   equivalent to what the Python code produces with structlog.

6. The task queue must be implemented with a proper min-heap (priority
   queue) data structure. Do NOT use Array.sort() as a substitute. The
   heap must order entries by (priority, insertion_counter) so that
   same-priority jobs come out in FIFO order, matching the Python
   heapq behavior.

7. All custom exception/error types from the Python source
   (InvalidJobStateError, QueueFullError, DuplicateJobError) must be
   translated to custom Error subclasses in TypeScript, with the
   class name preserved as the .name property.

8. Translate all Python dataclasses to TypeScript classes with
   equivalent constructor signatures, property accessors, and
   serialization methods (toDict / fromDict). The serialized
   dictionary keys must use camelCase in TypeScript (e.g., jobId,
   createdAt, retryCount) — the Python source's snake_case keys are
   adapted to TypeScript convention at the serialization boundary.

9. Translate all Python enums to idiomatic TypeScript enums or const
   objects that preserve the same string/numeric values.

10. The configuration class must validate all constraints on
    construction, throwing errors for invalid values, matching the
    Python __post_init__ validation exactly. Configuration property
    names use camelCase (maxWorkers, maxQueueSize, defaultPriority,
    schedulingStrategy, storageBackend, storagePath, retryLimit,
    timeoutSeconds, logLevel).

11. Produce a valid package.json and tsconfig.json. The project must
    compile with tsc and be runnable with ts-node or equivalent.

12. Translate the serialization utilities to produce JSON strings that
    round-trip through deserialization to structurally equivalent
    objects.

13. Translate the validation utilities preserving exact validation
    logic — same rules for job names, payload keys, priority ranges,
    and tag sanitization. Error message strings produced by these
    utilities must match the Python source verbatim.

## What NOT to Translate

- Do NOT translate any test files. The target repository should
  contain zero tests.
- Do NOT translate Python-specific dev tooling (pyproject.toml,
  requirements.txt are replaced by package.json/tsconfig.json).

## Deliverable

A complete, compilable TypeScript project under /target/ that
faithfully reproduces the behavior of the Python source. The project
structure and filenames must follow the layout specified in
"Project Structure & File Naming" above. Class, variable, and function
names follow TypeScript conventions (PascalCase / camelCase). Use ES
module style imports.
